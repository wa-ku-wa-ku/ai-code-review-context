上下文工具层代码生成说明（最终版）

AI 仓库级代码评审系统 · 上下文处理模块

## 1. 项目目标

本模块是 AI 仓库级代码评审系统中的“上下文处理模块”。它不直接负责判断代码是否存在问题，而是负责把用户上传的 Python 仓库转化为可查询、可追踪、可被 Agent 使用的代码上下文工具层。

核心目标是：

解析 Python 仓库结构；

建立代码文件、代码符号和调用关系索引；

生成评审任务卡片；

向 Agent 提供按需查询源码、调用链和相关上下文的工具接口；

避免一次性向 Agent 塞入完整源码，降低 token 消耗并提升上下文准确性。

## 2. 项目技术栈

根据项目 README，当前系统采用 Python 后端技术栈，支持 API、CLI 和 Docker Compose 启动方式。

# 启动 API
uvicorn smart_code_review.main:app --reload --port 8080

# 运行示例 CLI
smart-review --payload examples/review_payload.json

# 使用 Docker Compose 启动 Redis、API 和 Worker
docker compose up --build

## 3. 模块边界

## 3.1 本模块负责

接收 zip 包或本地仓库目录；

解压仓库并过滤无关文件；

扫描 Python 文件和配置文件；

使用 Python AST 抽取函数、类、方法、import、decorator 和调用关系；

识别 FastAPI / Flask 路由入口；

构建 code_files、code_nodes、code_edges 等索引；

生成评审任务 task_card；

向 Agent 暴露代码查询工具。

## 3.2 本模块不负责

不负责判断代码是否存在漏洞；

不负责生成最终评审报告；

不负责前端展示；

不负责多 Agent 调度策略；

第一版不实现完整 MCP Server，只预留工具化封装能力。

## 4. 总体设计思路

本模块借鉴 CodeGraph 的代码图谱工具思想，但不直接复刻其通用 MCP 形态。CodeGraph 更偏向通用代码理解工具，而本模块面向 Python 仓库全量评审场景，重点是“任务驱动的上下文工具层”。

用户上传 Python 仓库 zip
  ↓
API 接收请求并创建评审任务
  ↓
Redis 分发任务给 Worker
  ↓
Worker 调用上下文模块解析仓库
  ↓
构建代码索引和调用关系图
  ↓
生成 repo_summary 和 task_card
  ↓
Agent 接收 task_card
  ↓
Agent 通过上下文工具按需获取源码和调用链
  ↓
Agent 输出评审结果

核心原则：

仓库摘要和任务卡片作为初始输入直接下发给 Agent；

Agent 工具只保留代码探索能力；

系统内部接口不暴露给普通评审 Agent；

上下文索引数据与业务数据逻辑分离。

## 6. 核心数据模型

## 6.1 CodeFile

用于记录仓库中的文件信息。

## 6.2 CodeNode

用于记录代码符号。

## 6.3 CodeEdge

用于记录代码关系。

## 6.4 ReviewTask

用于记录上下文模块生成的评审任务。

## 7. 数据库存储方案

数据库建议采用“业务数据与上下文索引数据逻辑分离”的方式。

## 7.1 主业务数据库

主系统数据库保存：

仓库上传记录；

评审任务状态；

Agent 评审结果；

最终报告；

系统日志。

## 7.2 上下文索引库

上下文模块单独维护代码索引库，保存 code_files、code_nodes、code_edges、review_tasks、context_usage。第一版建议每个仓库生成一个独立 SQLite 索引库：

data/
└── repos/
    └── repo_001/
        ├── source/
        ├── context_index.sqlite
        └── metadata.json

Redis 只用于任务队列、Worker 状态和临时缓存，不用于长期保存代码索引。

## 8. 初始化下发内容

以下内容不是 Agent 工具，而是在 Agent 执行任务前由调度器直接下发。

## 8.1 repo_summary

用于让 Agent 了解仓库基本情况。

{
  "repo_id": "repo_001",
  "framework": "FastAPI",
  "python_files": 38,
  "entrypoints": 12,
  "test_files": 8,
  "main_packages": [
    "app.api",
    "app.services",
    "app.repositories",
    "app.models"
  ]
}

## 8.2 task_card

用于告诉 Agent 当前要评审什么。

{
  "repo_id": "repo_001",
  "task_id": "task_api_login",
  "task_type": "entrypoint_review",
  "target": "POST /login",
  "seed_node_id": "node_auth_login",
  "priority": "high",
  "review_focus": [
    "authentication",
    "password_check",
    "token_generation",
    "input_validation"
  ]
}

普通评审 Agent 不需要主动调用 list_review_tasks，也不需要自己获取完整仓库摘要。

## 9. Agent 工具接口设计

本模块对 Agent 暴露的工具主要对标 CodeGraph 的 search、node、callers、callees、trace、explore、context 等工具。第一版建议保留 8 个核心工具。

## 9.1 search_symbol

作用：根据关键字搜索函数、类、方法、路由等代码符号。

输入：

{
  "repo_id": "repo_001",
  "query": "login",
  "symbol_types": ["function", "method", "route"],
  "limit": 10
}

输出：

[
  {
    "node_id": "node_auth_login",
    "type": "function",
    "name": "login",
    "qualified_name": "app.api.auth.login",
    "file_path": "app/api/auth.py",
    "start_line": 12,
    "end_line": 35
  }
]

## 9.2 get_node_detail

作用：根据 node_id 获取代码节点详情和源码片段。

输入：

{
  "repo_id": "repo_001",
  "node_id": "node_auth_login",
  "include_source": true
}

输出：

{
  "node_id": "node_auth_login",
  "type": "function",
  "qualified_name": "app.api.auth.login",
  "file_path": "app/api/auth.py",
  "start_line": 12,
  "end_line": 35,
  "signature": "def login(req: LoginRequest)",
  "decorators": ["router.post('/login')"],
  "source": "def login(req: LoginRequest):\n    ..."
}

## 9.3 get_callees

作用：获取某个函数向下调用的函数。

输入：

{
  "repo_id": "repo_001",
  "node_id": "node_auth_login",
  "depth": 2,
  "limit": 20
}

输出：

{
  "root": "node_auth_login",
  "callees": [
    {
      "from": "node_auth_login",
      "to": "node_user_authenticate",
      "to_name": "UserService.authenticate",
      "file_path": "app/services/user_service.py",
      "line": 18
    }
  ]
}

## 9.4 get_callers

作用：获取哪些函数调用了当前函数，用于分析入口来源和影响范围。

输入：

{
  "repo_id": "repo_001",
  "node_id": "node_user_authenticate",
  "depth": 1,
  "limit": 20
}

输出：

{
  "root": "node_user_authenticate",
  "callers": [
    {
      "from": "node_auth_login",
      "from_name": "app.api.auth.login",
      "file_path": "app/api/auth.py",
      "line": 18
    }
  ]
}

## 9.5 trace_call_chain

作用：追踪两个节点之间是否存在调用路径。

输入：

{
  "repo_id": "repo_001",
  "start_node_id": "node_auth_login",
  "end_node_id": "node_find_user",
  "max_depth": 4
}

输出：

{
  "found": true,
  "path": [
    {
      "node_id": "node_auth_login",
      "name": "app.api.auth.login",
      "file_path": "app/api/auth.py"
    },
    {
      "node_id": "node_user_authenticate",
      "name": "app.services.user_service.UserService.authenticate",
      "file_path": "app/services/user_service.py"
    },
    {
      "node_id": "node_find_user",
      "name": "app.repositories.user_repo.UserRepository.find_by_username",
      "file_path": "app/repositories/user_repo.py"
    }
  ]
}

## 9.6 explore_related_symbols

作用：围绕某个节点返回一组相关节点和关系图，减少 Agent 多次调用。

输入：

{
  "repo_id": "repo_001",
  "node_id": "node_auth_login",
  "depth": 2,
  "include_source": false
}

输出：

{
  "center": "node_auth_login",
  "nodes": [
    {
      "node_id": "node_auth_login",
      "name": "login",
      "file_path": "app/api/auth.py"
    },
    {
      "node_id": "node_user_authenticate",
      "name": "UserService.authenticate",
      "file_path": "app/services/user_service.py"
    }
  ],
  "edges": [
    {
      "from": "node_auth_login",
      "to": "node_user_authenticate",
      "type": "calls"
    }
  ]
}

## 9.7 get_related_context

作用：根据 task_id 返回推荐查看的节点和文件。该工具不直接返回大段源码，只返回推荐目标。

输入：

{
  "repo_id": "repo_001",
  "task_id": "task_api_login"
}

输出：

{
  "task_id": "task_api_login",
  "seed_node_id": "node_auth_login",
  "recommended_nodes": [
    {
      "node_id": "node_auth_login",
      "reason": "API entrypoint"
    },
    {
      "node_id": "node_user_authenticate",
      "reason": "direct callee"
    },
    {
      "node_id": "node_find_user",
      "reason": "database access"
    }
  ],
  "related_files": [
    {
      "file_path": "app/config.py",
      "reason": "possible auth/token config"
    },
    {
      "file_path": "tests/test_auth.py",
      "reason": "related test file"
    }
  ]
}

## 9.8 get_file_snippet

作用：根据文件路径和行号范围获取源码片段。

输入：

{
  "repo_id": "repo_001",
  "file_path": "app/api/auth.py",
  "start_line": 10,
  "end_line": 40
}

输出：

{
  "file_path": "app/api/auth.py",
  "start_line": 10,
  "end_line": 40,
  "source": "@router.post('/login')\ndef login(req: LoginRequest):\n    ..."
}

## 10. 系统内部接口

## 11. 典型调用流程

以登录接口评审任务为例：

调度器向 Agent 下发 repo_summary 和 task_card

Agent 调用 get_node_detail(seed_node_id)

Agent 调用 get_callees(seed_node_id, depth=2)

Agent 对关键 callee 继续调用 get_node_detail

Agent 调用 get_related_context(task_id)

Agent 按需调用 get_file_snippet 获取局部源码

Agent 基于获取到的上下文完成评审

系统记录 context_usage，用于覆盖率统计

## 12. 第一版实现范围

第一版需要实现：

## 13. 验收标准

能成功解析一个 Python 示例仓库；

能生成 code_files、code_nodes、code_edges；

能识别 FastAPI / Flask 路由入口；

能生成基础 review task 和 task_card；

Agent 可以通过 get_node_detail 获取入口函数源码；

Agent 可以通过 get_callees / get_callers 查询调用关系；

Agent 可以通过 get_file_snippet 按文件路径和行号获取源码片段；

工具返回格式稳定，全部为 JSON；

上下文索引数据可以按 repo_id 独立管理；

Redis 只用于队列和缓存，不作为代码索引长期存储。

## 14. 给 AI 编码工具的实现要求

保持模块职责单一，不把评审逻辑写进上下文模块；

所有工具接口返回 JSON serializable 对象；

AST 解析失败时不能中断整个仓库解析，应记录错误并继续处理其他文件；

zip 解压必须防止路径穿越；

文件扫描必须跳过 .git、.venv、__pycache__、dist、build、site-packages 等目录；

node_id 和 edge_id 需要稳定生成，建议基于 repo_id、file_path、qualified_name 和 line number；

第一版先保证可运行、可测试、结构清晰，不追求动态调用关系完全准确。

## 15. 补充修订说明

本节是在原“上下文工具层代码生成说明（最终版）”基础上的补充说明，不推翻原设计，只进一步明确 Python 专项路线、评审任务含义和 LLM 边界。

### 15.1 Python 专项解析路线

- 第一版只面向 Python 仓库，核心解析方式采用 Python ast。
- Python ast 足够完成函数、类、方法、import、decorator、行号、签名和基础调用表达式抽取。
- 后续增强重点不是多语言，而是 Python 专项能力：import 解析、对象方法调用解析、FastAPI / Flask 路由规则、Pydantic / SQLAlchemy / pytest 等常见生态规则。
- 第一版暂不引入 tree-sitter 多语言体系，也不强制引入 astroid、libcst、pyright 等高级解析工具。

### 15.2 ReviewTask 的定义

ReviewTask 是系统生成给评审 Agent 的代码评审任务卡，不是工程师开发任务，也不是 Jira 实现任务。它用于限定 Agent 本轮评审的对象、起点、重点和相关上下文。

```json
{
  "task_id": "task_route_login",
  "repo_id": "repo_001",
  "task_type": "entrypoint_review",
  "target": "POST /login",
  "seed_node_id": "node_app_api_auth_login",
  "priority": "high",
  "review_focus": ["输入校验", "身份认证", "密码校验", "Token 生成", "异常处理"],
  "status": "pending"
}
```

### 15.3 LLM 边界

- 任务生成第一版不依赖 LLM，而是根据路由、模块、配置文件和未覆盖文件等规则生成。
- 任务分发不属于上下文模块核心职责，由调度器根据 status、priority 和任务类型完成。
- LLM 只出现在评审 Agent 执行阶段，用于根据代码证据判断风险、解释问题和生成建议。
- 上下文模块应保持确定性和可测试性，重点提供结构化证据，而不是直接判断代码是否存在漏洞。

### 15.4 任务覆盖策略

- 结构覆盖：每个 API 路由生成 entrypoint_review，每个重要模块生成 module_review，每个配置文件生成 config_review。
- 风险角度覆盖：不同任务类型绑定固定 review_focus，例如认证、输入校验、越权、文件上传、SQL 注入、敏感配置等。
- 覆盖率追踪：记录 Agent 通过工具访问过的 node 和 file，生成 coverage_report。
- 补充任务：对未被任何任务覆盖的源码文件生成 uncovered_file_review。

### 15.5 第一版优先级调整

1. 先完成文件扫描、AST 解析、SQLite 索引和基础查询工具，形成 Python 代码图谱最小闭环。
2. 再完成调用关系、路由入口识别、get_callees / get_callers 等图查询能力。
3. 再生成 repo_summary、review_tasks、task_card 和 get_related_context。
4. 最后补充 context_usage、coverage_report 和未覆盖文件任务。
5. MCP Server、多语言解析、向量库、复杂动态调用分析、任务优先级智能规划均作为后续增强。

### 15.6 给 AI 编码工具的额外要求

- 生成代码时优先保证上下文模块的确定性，不要在任务生成和分发中强依赖 LLM。
- ReviewTask、repo_summary、tool output 均应保持 JSON serializable。
- 调用关系无法精确解析时，应保留 unresolved call 信息，而不是丢弃。
- 工具接口应自动记录 context_usage，但不要求 Agent 主动维护覆盖率。
- 第一版目标是可运行、可测试、可演示，不追求 Python 动态调用的完全准确。
