# 上下文处理模块阶段拆解计划
AI 仓库级代码评审系统 · Python 上下文模块 · 工程交付版
## 一、文档目标
本文将上下文处理模块从 0 开始拆解为多个阶段。每个阶段都对应一个可验收的小目标，每个任务都包含任务说明、可见产物、验收标准和建议 Jira 类型，便于工程师按周期开发，也便于老师查看阶段性成果。
## 二、模块边界
- 输入：用户上传或指定的 Python 仓库。
- 处理：仓库扫描、AST 解析、代码索引、调用关系、路由入口、任务生成、工具接口、覆盖率追踪。
- 输出：repo_summary、review_tasks、task_card、上下文工具查询结果、coverage_report。
- 不负责：最终漏洞判断、最终评审报告、多 Agent 调度策略、前端展示、完整 MCP Server。

## 三、核心原则
- 先做代码图谱底座，再做评审任务层。
- 第一版只面向 Python，使用 Python ast 作为核心解析方式。
- 任务生成和任务分发第一版不依赖 LLM；LLM 只负责具体代码评审判断。
- 每个阶段都必须有可运行、可展示、可测试的交付结果。

## 四、阶段总览
| 阶段 | 阶段名称 | 可验收目标 |
|---|---|---|
| 阶段 0 | 工程骨架搭建 | 项目能启动，pytest 能运行，目录职责清楚。 |
| 阶段 1 | 仓库输入与文件扫描 | 能读取仓库，过滤无关目录，并扫描有效 Python 文件。 |
| 阶段 2 | Python AST 符号解析 | 能抽取函数、类、方法、import、decorator、签名和行号。 |
| 阶段 3 | SQLite 索引存储 | 能生成 code_files、code_nodes、code_edges，并按 repo_id 隔离。 |
| 阶段 4 | 调用关系与路由识别 | 能识别 FastAPI / Flask 入口点和基础调用关系。 |
| 阶段 5 | 上下文工具接口 | Agent 能通过工具查询符号、源码、文件片段和调用链。 |
| 阶段 6 | 评审任务生成 | 能生成 repo_summary、review_tasks、task_card 和 related_context。 |
| 阶段 7 | 覆盖率追踪与整体验收 | 能记录上下文使用情况，输出 coverage_report，完成端到端验收。 |

## 六、评审任务是什么

本文中的“任务”指系统生成给评审 Agent 的代码评审任务卡，不是工程师在 Jira 上领取的开发任务。它用于说明 Agent 当前要审什么、从哪个节点开始、重点关注哪些风险、需要关联哪些文件。

```json
{
  "task_id": "task_route_login",
  "repo_id": "repo_001",
  "task_type": "entrypoint_review",
  "target": "POST /login",
  "seed_node_id": "node_app_api_auth_login",
  "priority": "high",
  "review_focus": ["输入校验", "身份认证", "密码校验", "Token 生成", "异常处理"],
  "status": "pending"
}
```

## 七、任务生成与 LLM 的关系

- 任务生成：由上下文模块根据索引和规则完成，不调用 LLM。
- 任务分发：由调度器根据 status 和 priority 分发，不属于上下文模块核心职责。
- 任务执行：由评审 Agent 调用上下文工具，并由 LLM 判断代码风险。
- 覆盖保障：规则负责结构覆盖和风险角度覆盖，LLM 负责具体问题判断。

## 八、ReviewTask 代码示例

```python
from dataclasses import dataclass, field
from typing import List, Optional

@dataclass
class ReviewTask:
    """
    评审任务卡片。

    注意：
    这个任务不是开发任务，而是给评审 Agent 执行的代码评审任务。
    每个任务对应仓库中的一个评审目标，例如 API 入口、模块或配置文件。
    """

    task_id: str                 # 任务唯一 ID
    repo_id: str                 # 仓库 ID
    task_type: str               # entrypoint_review / module_review / config_review / uncovered_file_review
    target: str                  # 评审目标，例如 "POST /login" 或 "app/config.py"
    seed_node_id: Optional[str]  # 起始代码节点，例如 login 函数；配置文件任务可以为空
    priority: str                # high / medium / low
    review_focus: List[str]      # 本任务建议关注的评审角度
    related_files: List[str] = field(default_factory=list)  # 推荐关联文件
    status: str = "pending"      # pending / running / completed / failed
```

## 九、验收口径总结

- 阶段 1：我能找到代码。
- 阶段 2：我能理解代码结构。
- 阶段 3：我能存下代码图谱。
- 阶段 4：我能知道代码关系。
- 阶段 5：我能把图谱能力暴露给 Agent。
- 阶段 6：我能把仓库拆成评审任务。
- 阶段 7：我能证明哪些代码被覆盖。
